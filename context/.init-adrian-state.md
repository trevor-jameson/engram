## Decisions

### Product
- **Project name: `engram`** (memory-trace term from Trevor's notes). All context files/docs use "Engram."
- **Vision beyond v1**: not only a flashcard app — flashcards are the *first feature* of a lifelong continuous-learning system; long-term goal is supporting an autodidact habit/mindset. Context files should frame v1 scope accordingly (flashcard leg first, system extensible later).
- One-sentence: A local-first spaced-repetition app that runs as a thin scheduler over a folder of plain markdown files (one card per file, YAML review state), living inside an Obsidian vault — private and greppable.
- Single user (Trevor), personal use, forever — no multi-user, no auth, no accounts.
- Privacy is core motivation (left Anki over it): **no network calls, no telemetry, no cloud sync** (vault sync is user's concern, not the app's).
- Spaced repetition is a keystone habit → protecting the daily habit outranks clearing backlog; sessions must stay short and completable.
- User is 10-yr teacher + software engineer studying CS/math textbooks; cards are technical — must handle code blocks, ideally math notation.
- App is the flashcard leg of a larger study protocol (recaps + practice problems happen outside the app).
- **Long-form commitments doc written**: `context/learning-principles.md` (user-requested seventh context file; sourced from `../claude/learning-notes/`). Overview + CLAUDE.md must reference it; drafters should reference, not restate, it.
- Scientific commitments: (1) cue-indexing / encoding specificity — fronts are situations, not terms; word→definition is deliberate exception; (2) harvest, don't predict — cards sourced from demonstrated failures; (3) successive relearning — session opens with delayed free recall of previous session; (4) interleaving — queue shuffles across sources, never topic-blocked; (5) desirable difficulty with feedback — chronic failure = bad indexing → leech rewrite flow.
- Daily session flow: 1) free-recall prompt ("Last session: {sources} — brain dump before reviewing"); 2) review queue, capped ~20–30, interleaved, overflow carries to tomorrow by overdue-ness, never guilt-trip; 3) inbox triage at session end — capture→card conversion via card-type templates (situation-shaped = path of least resistance).
- Inbox capture: always-available, near-zero-friction one-line raw text action, decoupled from authoring.
- Metrics (minimal): retention rate per box; leech list; optional "fired in real life" tap.
- Free-recall mechanics: typed in-app; ungraded (for now — no scheduler input); saved to a session-log file in the vault so triage can mine it for gaps.
- Queue: cap 25/day (cap = review-queue size; creation unbounded). Floor-as-fill 5: if <5 due, pull forward soonest-due cards; never an obligation, no guilt mechanics. 0-due day still runs free recall + inbox triage.
- Queue selection: due cards by overdue-ness (most overdue first) up to cap; then order so no two consecutive cards share `source`, ties random.
- Weighting: `type: problem` cards count 2 against the cap (worked-problem retrieval is slow); all other types count 1. No separate math flag.
- Math rendering: **v1**. `$...$` / `$$...$$` in card bodies (same syntax Obsidian renders natively); app renders with locally-bundled KaTeX (no network). Files remain plain markdown.
- **v1 scope** (top 3 = first three): 1) daily session flow (free recall → review queue → inbox triage); 2) card CRUD over vault folder (md+YAML parse/write, type-template creation, inbox capture); 3) softened Leitner scheduler (boxes, demotion, floor-as-fill, problem weighting, leech flagging); 4) leech rewrite flow (surface flagged cards, prompt situation-shaped rewrite) — user explicitly pulled into v1; 5) markdown + code blocks + KaTeX rendering. Leech trigger confirmed: 4+ lifetime lapses (user briefly said consecutive, then reverted to lifetime).
- **v1 out-of-scope**: auth/accounts/multi-user/network/telemetry/cloud sync (forever, not just v1); metrics dashboard (retention-per-box) → v2 (leech *list* effectively in v1 via rewrite flow); "fired in real life" tap (aspirational, unscheduled); free-recall grading/scheduler feedback (deferred "for now"); mobile polish; images in cards; Anki import; multiple decks / source-based archival (v1 lays groundwork via required `source` field).

### Technical
- Form factor: **local web app** — localhost server reading the vault folder, sessions in browser (decided this session).
- Stack: **TypeScript end-to-end — Node server + React/Vite frontend.** Why: one language across server/UI; richest markdown/YAML/KaTeX ecosystem (gray-matter, remark/unified, KaTeX are native citizens). (Server framework + md/YAML libs to be pinned during drafting: proposal = Express or Hono, gray-matter, remark, KaTeX — confirm in Phase 2 finalize.) Package manager: **pnpm** (not npm).
- Card model: one card per md file; YAML frontmatter = review state; body = Q/A. Files are single source of truth; human-editable, Obsidian-linkable, greppable; any index/cache rebuildable from files.
- Reference schema fields (names negotiable, semantics not): box (1–7), due, lapses, created, source, type.
- Card types: symptom→cause, decision→trade-off, prediction, problem (front=spec, back=worked solution; first-class for math), definition (deliberate only).
- Scheduler: softened Leitner, intervals 1,2,4,8,16,32,64 days; pass → up one box; lapse → down one box (not reset); leech = 4+ lifetime lapses → rewrite flow, removed from rotation until rewritten/deleted.
- Auth: none (single local user).
- Repo boundaries (confirmed): `server/` (Node; sole owner of vault file I/O) with `vault/` (md+YAML read/write, card parsing, session-log writer), `scheduler/` (pure functions: Leitner, queue building, leech detection), `api/` (localhost-only HTTP routes); `web/` (React/Vite UI — talks only to API, never filesystem, never sees paths, only card IDs); `shared/` (types: Card, SessionLog, queue DTOs); `context/` (context files).
- Vault path: configurable to any filesystem path (env var/config file). v1 layout: **single deck** — cards in one subfolder (e.g. `vault/flashcards/`), session logs in `logs/` beside them.
- Multiple decks: **out of scope for v1**. Underlying goal is *archival* — retiring all cards from a source when it no longer fits the study plan (e.g. dropping a math textbook). v1 supports this future by tagging every card with `source`; a post-v1 archival feature will move cards by source out of circulation.
- **Invariants (CONFIRMED by user, all six):**
  1. Card files are sole source of truth; all app state rebuildable from folder.
  2. Every file the app writes remains valid, human-readable markdown Obsidian renders.
  3. Zero network requests.
  4. Daily session always completable in ≤ ~15 min (cap enforces).
  5. Scheduler never deletes/rewrites card content — only frontmatter state; content changes are user actions.
  6. Every card has a `source`; no card-creation path omits it (archival + interleaving key).

### Design (APPROVED)
- Aesthetic: calm, content-first, functional. MUI component library + theme. Follows system light/dark mode (two themes via `createTheme`).
- Palette tokens (MUI slots, light / dark):
  - `background.default` #FAFAF7 / #16181D (app canvas)
  - `background.paper` #FFFFFF / #1F2229 (card surfaces, dialogs)
  - `text.primary` #1F2328 / #E8E6E1 (card content)
  - `text.secondary` #57606A / #9AA0A8 (metadata: source, box, due)
  - `primary.main` #4A5D8A / #8C9EC9 — muted slate-indigo (actions, focus, links)
  - `success.main` #2E7D5B / #5BAE8C — Pass grading
  - `error.main` #B3574D / #C97B72 — Lapse grading (softened; lapsing is normal, not alarm-red)
  - `warning.main` #B58A3C / #C9A35E — leech flags
  - `divider` #E6E4DF / #2E323A
- Typography: UI chrome = system font stack at MUI defaults; card content = rendered markdown at 1.125rem/1.7; code = ui-monospace on paper-offset tint; KaTeX matches text size.
- Radius: `shape.borderRadius: 8`; cards/dialogs 12; chips full. Shadows ≤ MUI elevation 2 (flat-ish, calm).
- Layout: single centered column max 720px; one-thing-at-a-time screens (recall → card → card → triage); no sidebar/dashboard chrome; inbox capture in persistent minimal header; metadata small/secondary below card.

### Process (APPROVED)
- Workflow rules:
  1. Read all six context files at session start; update progress-tracker.md before ending any work session.
  2. One build unit at a time; no new unit while current is unverified.
  3. `server/scheduler/` stays pure — no I/O imports; changes there require unit tests covering box transitions, cap/floor, weighting, leech detection.
  4. Never write to the real vault during development — file-I/O tests use fixture folders; dev server points at a scratch vault.
  5. pnpm only; new dependencies require stated justification in tracker (bias: zero-dep, thin-scheduler ethos).
  6. File-format changes (frontmatter schema, log format) require explicit user sign-off — files outlive the app.
- Protected files (never edit unasked): `context/*` except progress-tracker.md; anything inside a configured vault path; .env/config holding the vault path.
- Verification per unit: typecheck + lint; unit tests pass; invariants hold (incl. card-file parse→write round-trip byte-stable beyond intended frontmatter changes); manually drive affected flow in browser against scratch vault.
